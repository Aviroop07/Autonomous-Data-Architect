### SEMANTIC ANALYSIS
When reading the description, you must carefully distinguish between:
1. **Table Names**: Entities that represent a collection of objects or events.
2. **Column Names/Features**: Attributes or properties that belong to a specific table.
3. **Data Behaviour/Modeling Constraints**: Rules about how data flows, evolves, or is restricted.
Analyze carefully whether a certain keyword actually warrants its own table or is simply a feature/column of an existing one.

- **Consolidation Principle (EER Analysis - CRITICAL)**:
    - **Generalization (Supertype)**: If multiple entities represent roles or categories of the same underlying physical entity and share >80% of their attributes, you **MUST** consolidate them into a single **Supertype** (e.g., a generic category name) with a discriminator column.
    - **BANNED**: Do NOT split entities by role if they describe the same kind of physical actor.
    - **Specialization (Subtype)**: Only create specialized separate tables if a subclass has at least 3 unique attributes OR a fundamentally distinct relational life-cycle.
- **Dimension Separation**: DO NOT collapse distinct dimension categories into a single generic table if they are fundamentally different. However, follow the Generalization rule above for similar human/actor roles.
- **No Over-Engineering**: Do not add "common" tables or standard entities if they are not requested in the input.
- **Strict Columns**: Do not fabricate columns. Only include attributes mentioned in the description. If an entity is mentioned without specific attributes, include only its primary key and any necessary foreign keys.
- **Infer Structural Relationships**: Even if a relationship is not stated word-for-word, if a Dimension table clearly belongs to a Fact table in the same chunk, you MUST create the logical foreign key. Fact tables reference Dimension tables.
- **NO HALLUCINATION (CRITICAL)**:
    - You MUST NOT use your general knowledge of a domain to manufacture entities, columns, or relationships that are not explicitly present in the provided facts.
    - An `ANALYTICAL` or `BEHAVIORAL` fact (e.g., about fraud detection rates) does NOT justify the creation of a `PAYMENT` or `ORDER` table unless those entities are also explicitly mentioned as structural components in the chunk. **PHYSICALITY OVERRIDE applies only when the entity is mentioned in a `[STRUCTURAL]` fact.**
    - If a chunk contains only analytical or constraint facts without defining any entities, do NOT invent a schema. Return an empty `tables` list.
- **NO PARAMETER POLLUTION (CRITICAL)**:
    - You must NOT create tables that store statistical distribution parameters (e.g., "Mean", "Standard Deviation", "Weight"), modeling constants, or abstract rates.
    - If a fact describes a mathematical rule, a distribution, or a behavioral rate, it is considered "Flesh" and should NOT be materialized as a separate table.
    - **Business Measures (MANDATORY)**: Attributes and measures such as "Scores," "Rates," "Ratios," "Durations," "Balances," "Principal," and "Amount" are core business attributes of an entity and **MUST** be materialized as physical columns in the entity's table (e.g., `credit_score` in `CUSTOMER`, `interest_rate` in `LOAN`, `principal` in `LOAN`).
    - **NO SINGLE-ATTRIBUTE METRIC TABLES**: You are strictly PROHIBITED from creating a separate table for a single metric (e.g., a `CREDIT_SCORE` table with only a score and a FK). These MUST be flattened into the primary actor or event table.
    - **Multi-Entity Mandate (CRITICAL)**: If the input facts mention multiple distinct business entities (e.g., "Applicants" and "Loans", or "Orders" and "Shipments"), you **MUST** create separate tables for each. Never collapse distinct event/transaction entities into an actor table.
    - **Entity Existence Check**: Before finalizing, verify that every primary noun in the facts that has its own unique ID or lifecycle (e.g., "Loan", "Application", "Transaction") has a corresponding table.
    - **COMPARISON TARGET MATERIALIZATION (CRITICAL)**:
    - If a fact describes a rule where an attribute is compared to a value (e.g., "Amount < $50K", "DTI > 0.4", "Age >= 18"), that attribute **MUST** be materialized as a physical column in the appropriate table.
    - **No Exceptions**: Even if the attribute is not explicitly tagged as `STRUCTURAL`, its presence in a rule makes it structurally mandatory for the integrity of the data generation.
    - If you are unsure which table it belongs to, assign it to the primary noun/entity mentioned in the same fact or the table it most logically describes (e.g. "Amount" belongs to `LOAN`, "Age" belongs to `PERSON`).
- **RULE-VARIABLE DISCOVERY**: Scan all `[LOGICAL]` and `[ANALYTICAL]` facts for any noun that appears to be a variable or state (e.g., "Status", "Stage", "Category"). These MUST be columns.
- **NUMERIC BLACKLIST (STRICT)**: You are strictly PROHIBITED from using `VARCHAR` or `STRING` for any column containing keywords like "Amount", "Principal", "Balance", "Price", "Cost", or "Rate". These MUST be `FLOAT` or `DECIMAL`.
- **NUMERIC RANGE OVERRIDE**: If an attribute is described by a numeric range (e.g., "300-850", ">= 0"), it **MUST** be mapped to a numeric type. `VARCHAR` is forbidden for these.
- **NUMERIC TYPE PRECISION**: When materializing these columns, use precise numeric types:
    - `INTEGER` for: counts, discrete scores, rankings, or ages.
    - `FLOAT` or `DECIMAL` for: rates, ratios, percentages, or statistical distributions.
    - **Avoid `VARCHAR`** for any attribute that implies a mathematical or statistical property.
- Only materialize "Skeleton" entities that represent the physical objects, events, or actors in the domain.
- The schema should represent the **Physical Reality** of the data collection, not the **Statistical Theory** or auxiliary parameters behind it.
- **MANDATORY MATERIALIZATION**: The following entities MUST be materialized as tables whenever they are mentioned in a chunk, regardless of their primary tag:
    - Transactions, Orders, Payments, Fulfillment, Shipments.
    - Metrics, Gauges, Counters, Spans, Traces, Logs.
    - Users, Customers, Products, Merchants.

### FK DIRECTION, CARDINALITY & ROLE-PLAYING (CRITICAL)
When establishing a relationship between two entities:
1. **Rule**: The Foreign Key ALWAYS goes in the **Child** table, pointing to the **Parent**.
2. **Cardinality Matrix (Reasoning Step)**: Evaluate every relationship using this flow:
   - **Identity Multiplicity**: Does a single record in the Parent table imply one, or multiple, related records in the Child?
   - **Peer Interaction vs. Hierarchy**: Is Table A a "container/owner" of Table B (Hierarchical)? Or are A and B independent entities interacting (Peer)?
   - **M:N Join Logic**: If the relationship is described as a "mapping," "connection," "linkage," or "intersection" where both sides can be plural (e.g. "Multiple A have multiple B"), you MUST produce a **JOIN TABLE** rather than a simple foreign key.
   - **BANNED: DUAL FOREIGN KEYS**: You are explicitly PROHIBITED from creating two direct foreign keys that reference each other (A.b_id and B.a_id). This is a common failure mode. For bi-directional relationships, always use a **JOIN TABLE**.
   - **No Structural Contradictions**: If you produce a JOIN TABLE for a Many-to-Many relationship (e.g., `PLAYER_GUILD`), you MUST NOT also create a direct 1:N Foreign Key for that same relationship (e.g., `guild_id` on `PLAYER`). A relationship must be modeled EXACTLY ONCE, either as a Foreign Key OR as a Join Table.
3. **Role-Based Generalization**: If an entity acts as a bridge or participant involving multiple occurrences of the same conceptual category, you MUST use distinct, **role-based foreign key names** reflecting their meaning in that specific context rather than a generic id.
4. **Participation Detection (Junction Tables)**: If a fact describes that 'members (plural) do X' and 'hours/counts/details per member per event are tracked', this implies a many-to-many relationship requiring a junction table: one table for the event, one for the individual participation record. Do NOT collapse these into a single table with a member_id FK.
5. **Structural Integrity Contract**: When you establish a relationship (A references B), you MUST explicitly ensure that the `referencing_column` is present in the `columns` list of the referencing table (A).
6. **Completeness Audit**: Before finalizing, match every `FACT` tagged with an attribute against your column list. You MUST NOT lose attributes (e.g. `start_time`, `end_time`).

3. **Multi-Tag Guidance**:
    - `STRUCTURAL`: Core table/column definition and relationships/FKs.
    - `LOGICAL`: Unique keys, non-null requirements, or check constraints. **If a `[LOGICAL]` fact names an entity and specifies a value range/format for an attribute (e.g., "Monthly price >= 0"), ADD the attribute as a physical column in that entity's table.**
    - `METADATA`: Information describing the system scale or global industry; use this for naming conventions and overall domain model alignment.
    - `STATISTICAL`: Generally ignore for physical schema, unless: (a) it implies a time-series column, (b) it implies a probabilistic event record, or (c) it contains keywords like "stored", "recorded", "logged", or "tracked".

### RESTRICTIONS
1. **Table Naming**: Use `UPPER_SNAKE_CASE` strictly. Names must be human-readable and ALWAYS SINGULAR. Do NOT include prefixes/suffixes like 'FACT', 'DIM', 'ID', 'ATTR', or 'TABLE' in the table name itself. No white spaces.
2. **Column Naming**: Use `lower_snake_case` strictly. Names must be human-readable. No white spaces.
3. **Relationship Consistency**: Foreign key names and references must be consistent with the actual table and column names being linked.
4. **Uniqueness**: Table names must be unique within the entire schema. Column names must be unique within a single table.
5. **No Dangling Foreign Keys**: A foreign key MUST reference a table that actually exists in your generated `tables` list. Do NOT fabricate references to tables you did not create in this segment.
6. Primary Keys:
    - Never use composite primary keys.
    - **Naming Convention**: If no explicit primary identifier is mentioned or hinted at in the facts, always use the naming convention `table_name_id` for your generated surrogate PK.
    - **Natural Keys vs. PKs**: If a fact mentions a globally unique natural key (e.g., "Airplane Number", "unique serial number"), and it is clearly the primary identifier of the entity, use it as the Primary Key.
    - **Surrogate Fallback**: If NO explicit unique identifier is provided for an entity, generate a surrogate PK in the `table_name_id` format.
    - If the input provides an `_id` column whose noun matches the table name, use it as the primary key.
    - If the `_id` column's noun does NOT match the table name, it is a foreign key column, NOT the primary key. Generate a surrogate key for the table instead in the format `table_name_id`.
    - Each table must have exactly one primary key, and it MUST be explicitly included in the `columns` list of the `Table` object. You MUST include it even if the table has no other attributes.
7. **Column Integrity**:
    - Any column used as a `referencing_column` in the `relationships` list MUST also be explicitly present in the `columns` list of the referencing table.
    - **FK Data Type Matching**: The data type of a Foreign Key column MUST exactly match the data type of the Primary Key column of the table it refers to. If the referred table's PK is `INTEGER`, the referencing FK column MUST also be `INTEGER`.
8. **Unique Constraints**:
    - **When to Apply**: Apply a UNIQUE constraint (singleton or composite) if a fact explicitly mentions uniqueness (e.g., "email must be unique") or if an attribute/set of attributes acts as a natural key in the domain (e.g., "each employee has a unique government ID", "a customer can only have one active subscription per product").
    - **No PK in Unique**: NEVER include the Primary Key column in any UNIQUE constraint. The PK is unique by definition. Including it in a composite unique constraint is redundant and violates architectural standards.
    - **No Subset Redundancy**: If a set of columns `{{A}}` is already unique, do NOT create another unique constraint on `{{A, B}}`. The larger set is a proper superset of an already unique key and is technically redundant. If you believe `{{A, B}}` is the correct unique set, you must remove `{{A}}` first.
    - **Modeling**: Use the `unique` list in the `Table` object. Each item in this list is a `CompositeUnique` object containing a list of `columns`.
9. **Foreign Keys**:
    - When defining a relationship, use exactly these field names: `referencing_table` (child), `referencing_column` (child FK), and `referred_table` (parent).
    - You do not need to specify the referred column; by default, it is the primary key of identical name/ID of the referred table.
10. **Minimum Content**: Each table must have at least one column (the primary key).
11. **Relationships**: In per-shard generation, a table is permitted to be isolated if its facts do not describe any relationship to other tables in the current shard. Do NOT invent foreign keys to satisfy the 'connected graph' requirement. The global merger handles cross-shard connectivity.
