\begin{abstract}
\textit{[Placeholder $\sim$150\,words. Motivate the greenfield database-generation problem: no real data,
no formal spec, only a natural-language description.
Introduce ScribbleDB as a four-stage agentic pipeline
(fact extraction $\to$ schema generation $\to$ constraint modeling $\to$ code generation)
whose output is a relational schema (DDL) plus an executable Python program that materializes
the populated tables.
Highlight the key new contributions versus prior workshop version:
(i)~deterministic inverse-function code generation for single-dependency DAG nodes,
(ii)~multivariate distribution support,
(iii)~ancestral sampling for logical constraints (FA = 1.0),
(iv)~context enrichment and shard-and-merge for scalability.
Close with headline numbers: $\sim$6$\times$ table accuracy and $\sim$4$\times$ lower latency
over SchemaAgent across three benchmark suites.]}
\end{abstract}

\section{Introduction}
% ============================================================

\textit{[Para 1 --- Why synthetic databases matter: application development, testing, benchmarking;
data-privacy and greenfield constraints prevent access to real data.]}

\textit{[Para 2 --- State of the art and its limitations:
ab-initio generators (MUDD, PDGF, Myriad) require formal declarative specs;
regeneration-based tools (DBSynth, RSGen, Hydra, SAM) require production artefacts.
Neither suits a non-technical domain expert with only an NL description.]}

\textit{[Para 3 --- LLMs as enabler and source of new challenges.
Strong NL-to-SQL~\cite{yu2019spider} and text-to-schema~\cite{wang2025text2schema} capability.
Three core challenges for full DB synthesis:
\textbf{(a)~Expressiveness} (complex cross-table constraints),
\textbf{(b)~Hallucination} (error propagation across stages),
\textbf{(c)~Scalability} (context window and materialization throughput).]}

\textit{[Para 4 --- ScribbleDB overview: how it addresses each challenge.
Emphasise that the \emph{output} is a DDL schema and an executable Python program,
not tables directly.]}

\medskip\noindent\textbf{Contributions.}
\begin{itemize}[leftmargin=*]
  \item End-to-end NL-to-database agentic pipeline; outputs relational DDL schema
        + executable Python code for table materialization.
  \item \textit{[New contribution]} Deterministic code generation for single-dependency
        columns in the DAG via inverse functions on join/aggregate tables,
        reducing LLM calls and guaranteeing correctness-by-construction.
  \item \textit{[New contribution, partially fleshed out]} Multivariate distribution
        support for cross-column correlations beyond univariate priors.
  \item Shard-and-merge schema generation with Gale--Shapley entity alignment.
  \item Ancestral sampling for logical constraint satisfaction; FA = 1.0 empirically.
  \item Comprehensive evaluation: $\sim$6$\times$ table accuracy, $\sim$4$\times$ lower
        latency vs.\ SchemaAgent~\cite{wang2025text2schema} across three benchmark suites.
\end{itemize}

\section{Problem Formulation}
% ============================================================

\textit{[Para 1 --- Input: NL description $\mathcal{D}$, target row count per fact table.
Output: (i)~relational schema $\mathcal{S}$ (table definitions, PKs, FKs, data types),
(ii)~Python program $\mathcal{P}$ s.t.\ executing $\mathcal{P}$ produces tables satisfying
$\mathcal{S}$ and all constraints implied by $\mathcal{D}$.]}

\textit{[Para 2 --- Correctness criteria: schema validity (PK uniqueness, FK referential
integrity, type correctness), constraint fidelity (FA = 1.0 for logical; KS/MRE/NLL within
thresholds for statistical), and completeness (recall of schema elements described in $\mathcal{D}$).
The system assumes $\mathcal{D}$ is internally consistent.]}

\section{Stage 4: Code Generation}
\label{sec:stage4}
% ============================================================

\subsection{Deterministic Inverse-Function Code for Single-Dependency Columns}
\label{subsec:det-codegen}
\textit{[Para: Key new contribution.
For each column node in $\mathcal{G}$ that has \emph{exactly one} dependency,
code is synthesized deterministically---no LLM call.
Strategy: (i)~synthesize an intermediate join or aggregate table satisfying the
aggregate constraint; (ii)~decompose back to base columns using an inverse function
whose form is determined by the aggregation type
(SUM $\to$ uniform partition; MAX $\to$ argmax selection; JOIN $\to$ FK assignment).
Guarantees correctness-by-construction; eliminates LLM API cost for these nodes.]}

\textit{[Para: The LLM code-generator agent is invoked only when column dependencies are
\emph{intertwined}---multiple constraints share overlapping column sets with no clean
closed-form inverse.
In that case, the agent follows the reverse-causal order of $\mathcal{G}$ and synthesizes
inverse functions generatively (ancestral sampling).]}
