\textit{[Placeholder $\sim$150\,words. Motivate the greenfield database-generation problem: no

\section{Introduction}
% ============================================================

\textit{[Para 1 --- Why synthetic databases matter: application development, testing,
benchmarking; data-privacy and greenfield constraints prevent access to real data.
Emphasise the greenfield setting: no existing deployment, no schema, no seed data.]}

\textit{[Para 2 --- State of the art and its limitations:
ab-initio generators (MUDD, PDGF, Myriad) require formal declarative specs;
regeneration-based tools (DBSynth, RSGen, Hydra, SAM) require production artefacts.
Neither suits a non-technical domain expert with only an NL description.]}

\textit{[Para 3 --- LLMs as enabler and source of new challenges.
Strong NL-to-SQL~\cite{yu2019spider} and text-to-schema~\cite{wang2025text2schema} capability.
Three core challenges for full DB synthesis:
\textbf{(a)~Expressiveness} (complex cross-table constraints),
\textbf{(b)~Hallucination} (error propagation across pipeline stages),
\textbf{(c)~Scalability} (context window degradation; materialization throughput).]}

\textit{[Para 4 --- ScribbleDB overview: how it addresses each challenge.
Emphasise that the \emph{output} is a DDL schema and an executable Python program, not
tables directly---separating correctness verification from data materialization.]}

\medskip\noindent\textbf{Contributions.}
\begin{itemize}[leftmargin=*]
  \item End-to-end NL-to-database agentic pipeline; outputs relational DDL schema
        + executable Python code for table materialization.
  \item \textit{[New]} Deterministic code generation for single-dependency columns
        in the DAG via inverse functions on join/aggregate tables,
        reducing LLM calls and guaranteeing correctness-by-construction.
  \item \textit{[New, partial]} Multivariate distribution support for cross-column
        correlations beyond univariate priors.
  \item Shard-and-merge schema generation with Gale--Shapley entity alignment.
  \item Ancestral sampling for logical constraint satisfaction; FA = 1.0 empirically.
  \item Comprehensive evaluation: $\sim$6$\times$ table accuracy, $\sim$4$\times$ lower
        latency vs.\ SchemaAgent~\cite{wang2025text2schema} across three benchmark suites.
\end{itemize}

\section{Background and Related Work}
% ============================================================

\subsection{Database Generation}
\textit{[Para: Ab-initio generators (MUDD~\cite{stephens2004mudd}, PSDG~\cite{hoag2007psdg},
PDGF~\cite{rabl2010pdgf}, Myriad~\cite{alexandrov2012myriad}) operate from declarative
specifications; powerful but require expert modeling effort.
Regeneration-based approaches (DBSynth~\cite{rabl2015dbsynth}, RSGen~\cite{shen2013rsgen},
Hydra~\cite{sanghi2018hydra}, SAM~\cite{yang2022sam}, DScaler~\cite{zhang2016dscaler},
TouchStone~\cite{wang2023scalable}) exploit statistical metadata or workload traces from an
existing deployment; inapplicable in greenfield settings.
Neither class accepts NL input or handles joint distributional and logical constraints.]}

\subsection{LLM-based Schema and SQL Generation}
\textit{[Para: NL-to-SQL benchmarks (Spider~\cite{yu2019spider}) and the NL-to-SQL
survey~\cite{luo2025nl2sql} demonstrate strong LLM capability on structured-output tasks.
SchemaAgent / Text2Schema~\cite{wang2025text2schema} is the closest prior work:
it converts NL to a relational schema but stops short of data materialization, distributional
constraints, and logical fidelity.
ScribbleDB is the first system to cover the full synthesis pipeline.]}

\subsection{Agentic Systems for Structured Output}
\textit{[Para: Multi-agent orchestration, self-repair loops, and tool-augmented LLMs have been
applied to code generation, document understanding, and data engineering.
ScribbleDB adopts these patterns but adds deterministic guard rails and a correctness-by-construction
materialization scheme, distinguishing it from generation-only agentic approaches.]}

\section{Problem Formulation}
% ============================================================

\noindent\textbf{Definition 1 (Database Synthesis Problem).}
\textit{[Formal definition box. Input: an NL description $\mathcal{D}$ of a target database
domain plus a target row count $n$ per fact table.
Output: (i) a relational schema $\mathcal{S}$ (table definitions, primary keys, foreign keys,
data types), and (ii) a Python program $\mathcal{P}$ such that executing $\mathcal{P}$
produces a set of populated CSV/SQL tables satisfying $\mathcal{S}$ and all constraints
implied by $\mathcal{D}$.
Assumption: $\mathcal{D}$ is internally consistent (no contradiction detection).]}

\medskip
\noindent\textbf{Correctness Criteria.}
\textit{[Para: (a) Schema validity: PK uniqueness, FK referential integrity, data-type
correctness.
(b) Statistical fidelity: generated column distributions match those implied by $\mathcal{D}$
(measured by KS, MRE, NLL).
(c) Logical fidelity: all decision-tree and aggregation constraints satisfied
(measured by FA).
(d) Completeness: recall of schema elements described in $\mathcal{D}$.]}

\medskip
\noindent\textbf{Running Example.}
\textit{[Para: Introduce the \emph{Credit Risk (Loan Origination)} database as the running
example throughout the paper.
Summarise the NL description: applicants with credit scores (300--850), annual income, DTI
ratio; approved loans with an interest rate derived from a tiered lookup of credit score and
loan amount (e.g., credit score $>$ 750 \& amount $<$ \$50K $\Rightarrow$ rate $= 3.5\%$);
overall interest rate follows a Gaussian distribution.
This example is used in all stage-level figures and the case study in \S\ref{sec:eval}.]}

% ============================================================

\textit{[Para 1: Four-stage pipeline overview: Fact Extraction $\to$ Schema Generation $\to$
Constraint Modeling $\to$ Code Generation.
Each stage uses deterministic guard rails; failures trigger a bounded repair loop

\section{Stage 1: Fact Extraction}
\label{sec:stage1}
% ============================================================

\textit{[Intro sentence: Stage 1 transforms the raw NL description $\mathcal{D}$ into a
structured, enriched, and classified set of atomic facts $\mathcal{F}$ via a pipeline of
LLM agents interleaved with deterministic validation and filtering steps.]}

\subsection{Atomic Fact Decomposition}
\textit{[Para: A \textsc{FactExtractor} LLM agent (tool-augmented: DuckDuckGo web-search)
decomposes $\mathcal{D}$ into candidate \emph{atomic facts}---minimal single-requirement
statements (e.g., ``An applicant has a credit score between 300 and 850'').
Each fact carries: a unique identifier, its verbatim text, a list of
\texttt{referenced\_fact\_ids} (other facts it depends on or elaborates), an
\texttt{is\_external} flag, and an origin span anchoring it to a substring of $\mathcal{D}$.
The agent tracks previously errored fact IDs across repair iterations to avoid re-introducing
known invalid facts.]}

\subsection{Deterministic Fact Validation}
\label{subsec:det-fact-val}
\textit{[Para: After each extraction iteration the candidate set is subjected to a
\emph{deterministic validator} that performs five correctness checks entirely without LLM
involvement (Algorithm~\ref{alg:det-fact-val}).
Errors are typed (CRITICAL / MEDIUM / LOW) and fed back to the \textsc{FactExtractor} agent
as a structured repair signal.
Only CRITICAL errors trigger a re-extraction; MEDIUM and LOW errors are surfaced as warnings.
Short origin spans ($<$10 characters) are downgraded to LOW rather than CRITICAL to avoid
over-flagging terse but correct facts.]}

% Algorithm 1: Deterministic Fact Validator (sub-step)
\begin{algorithm}[t]
\caption{Deterministic Fact Validator}
\label{alg:det-fact-val}
\begin{algorithmic}[1]
  \Require Candidate fact set $F$, source NL text $\mathcal{D}$
  \Ensure List of typed error records $E$
  \State \textit{[Placeholder: normalize refs; check (1) invalid referenced IDs,
         (2) self-references, (3) DFS cycles on reference graph,
         (4) unverifiable origin via FactOriginMatcher (short spans $\to$ LOW severity),
         (5) external facts without any cited non-external fact;
         return typed errors CRITICAL / MEDIUM / LOW]}
\end{algorithmic}
\end{algorithm}

\subsection{Integrity Verification}
\textit{[Para: Once the fact set passes deterministic validation, a \textsc{Verifier} LLM agent
performs a semantic audit.
It inspects the fact set for: (a)~\emph{missing information} (entities or constraints in
$\mathcal{D}$ not yet captured as facts), (b)~\emph{introduced information}
(facts that cannot be grounded in $\mathcal{D}$), (c)~\emph{changed constraints}
(paraphrase drift from the original wording), and (d)~\emph{unresolved ambiguities}.
The verifier emits a structured \texttt{IntegrityReport}; if non-empty, the
\textsc{FactExtractor} agent is re-invoked with the report appended to its context.
This loop continues until the report is empty or the retry budget is exhausted.]}

\subsection{Fact Classification}
\textit{[Para: After the verification loop, a \textsc{Tagger} LLM agent performs a
\emph{one-shot} pass over the verified fact set.
Each fact is tagged with one or more of four types:
\textsc{Structural} (entity definitions, FK relationships),
\textsc{Logical} (if--then rules, conditional constraints),
\textsc{Statistical} (distributional parameters, value ranges), and
\textsc{Metadata} (documentation, provenance notes).
These tags are stored in the \texttt{tags} field of each \texttt{AtomicFact} and guide
downstream agents: Schema Generation prioritises \textsc{Structural} and \textsc{Logical}
facts; Code Generation consumes \textsc{Statistical} facts for distribution sampling.]}

\subsection{Underspecification Detection and Context Enrichment}
\textit{[Para: A deterministic \emph{underspecification detector} counts entities ($<$4),
relationships, constraints, and total facts ($<$8) in the verified set and emits
domain-specific DuckDuckGo search queries when the schema is likely under-described.
These queries supplement those derived by the \textsc{ContextEnricher} agent.]}

\textit{[Para: The \textsc{ContextEnricher} runs in two phases.
\textbf{Phase 1 (pre-search)}: extract acronyms via regex $[A-Z]\{2,5\}$, combine with any
suggestions from the \texttt{IntegrityReport} (up to 5 queries total), and execute DuckDuckGo
searches via \texttt{prefetch\_and\_format\_searches}; results are formatted as grounding
context.
\textbf{Phase 2 (LLM)}: the agent receives the enriched context and the verified fact set
and generates a list of \emph{external facts} (technical definitions, architecture patterns,
domain constraints) to augment $\mathcal{F}$.
A \textsc{ContextAuditor} LLM agent then reviews the proposed external facts, accepting or
rejecting each with a reason; accepted facts are accumulated across iterations.
Ablation shows $+10\%$ Table F1 from enrichment alone (\S\ref{sec:eval}).]}

\subsection{External Context Filtering}
\textit{[Para: The accepted external facts are passed through a deterministic
\emph{external context filter} (Algorithm~\ref{alg:ext-filter}) before being merged into
$\mathcal{F}$.
The filter deduplicates, validates cross-references, and assigns a structured
\texttt{ExternalFactKind} label to each accepted fact via keyword matching.
The kind label informs how Schema Generation and Code Generation agents interpret and
weight the fact.]}

\begin{algorithm}[t]
\caption{External Context Filter}
\label{alg:ext-filter}
\begin{algorithmic}[1]
  \Require Accepted external facts $F_{\text{ext}}$, original non-external facts $F_{\text{orig}}$
  \Ensure Classified, deduplicated external fact set $F_{\text{ext}}^{*}$
  \State \textit{[Placeholder: (1) dedup by normalised text; (2) discard self-referencing facts;
         (3) discard facts referencing IDs not in $F_{\text{orig}}$;
         (4) classify each fact into ExternalFactKind via keyword matching

\end{algorithm}

\section{Stage 2: Schema Generation}
\label{sec:stage2}
% ============================================================

\textit{[Intro sentence: Stage 2 maps the enriched fact set $\mathcal{F}$ to a globally
consistent relational schema $\mathcal{S}$ via a shard-and-merge strategy augmented with
domain intelligence, deterministic validation, and a multi-agent review loop.]}

\subsection{Domain Intelligence Extraction}
\textit{[Para: Before fact clustering, a \textsc{DomainIntelligenceExtractor} one-shot LLM
agent is invoked with the domain name and the analytical goal implicit in $\mathcal{F}$.
It produces a \texttt{DomainIntelligenceReport}---a structured summary of typical entity
types, common relationship patterns, and domain-specific design conventions for the target
domain (e.g., normalisation conventions in healthcare, regulatory FK structures in finance).
This report is passed as additional context to the \textsc{SchemaArchitect} agents in each
shard and to the \textsc{DomainAuditor} loop (§\ref{subsec:schema-finalize}), anchoring the
schema in domain expectations rather than solely in the facts.]}

\subsection{Fact Clustering and Parallel Shard Generation}
\textit{[Para: A \textsc{Chunker} LLM agent partitions $\mathcal{F}$ into a \texttt{ChunkedPlan}:
a list of \emph{chunks} (each a labeled cluster of fact IDs covering one cohesive sub-domain)
plus a \texttt{core\_modeling\_facts} set of globally applicable facts (e.g., cross-table
FK conventions, domain-wide cardinality rules) that are broadcast to every shard.
The Chunker participates in a deterministic repair loop: structural errors in the proposed
plan (e.g., orphaned fact IDs, empty chunks) are fed back as typed error signals.]}

\textit{[Para: Each chunk is processed in parallel by a \textsc{SchemaArchitect} LLM agent
that receives the chunk's facts, the \texttt{core\_modeling\_facts}, and the
\texttt{DomainIntelligenceReport}.
It produces a candidate schema shard: table definitions, columns, PKs, and intra-shard FKs.
A deterministic repair loop (Algorithm~\ref{alg:shard-val}) guards each shard; the agent
maintains a \texttt{fix\_history} to avoid re-introducing previously repaired errors.]}

\subsection{Deterministic Shard Validation}
\textit{[Para: Each shard undergoes a pure-algorithmic check before merging:
PK uniqueness within the shard, FK referential completeness (all FK targets exist in the shard
or are flagged as cross-shard pending), and schema-graph connectivity (no isolated table).
Failures emit a typed repair signal to the schema-architect agent; no LLM call within
the validator itself.]}

% Algorithm 4: Deterministic Shard Validation (sub-step)
\begin{algorithm}[t]
\caption{Deterministic Shard Validation}
\label{alg:shard-val}
\begin{algorithmic}[1]
  \Require Schema shard $S_i$
  \Ensure Validated $S_i$ or typed repair signal
  \State \textit{[Placeholder: (1) PK uniqueness per table (PK\_CLASH);
         (2) FK target existence or cross-shard pending flag (FK\_DANGLE);
         (3) BFS connectivity---no isolated table (ISOLATED\_TABLE);
         emit typed repair signal to SchemaArchitect on failure]}
\end{algorithmic}
\end{algorithm}

\subsection{Shard Merging via Stable Matching}
\textit{[Para: Independently generated shards may contain overlapping representations of the
same entity.
The \textsc{SchemaMerger} is a fully deterministic three-pass procedure
(Algorithm~\ref{alg:merge}) that identifies and resolves these overlaps.]}

\textit{[Para: \textbf{Similarity scoring.}
For each candidate table pair $(T_a, T_b)$ across shards, a composite score
$\sigma = \alpha \cdot \sigma_{\text{name}} + (1-\alpha) \cdot \sigma_{\text{attr}}$ is
computed. $\sigma_{\text{name}}$ is the cosine similarity of sentence-transformer embeddings
(all-MiniLM-L6-v2) with a lexical fallback: $\sigma_{\text{lex}} = 0.3 \cdot J + 0.7 \cdot O$
(Jaccard $J$, context-overlap $O$); the final name score is $\max(\sigma_{\text{lex}},
\sigma_{\text{emb}})$, using the strongest available signal.
$\sigma_{\text{attr}}$ is a second Gale--Shapley~\cite{gale1962} pass over column pairs.
Shard pairs with $\sigma > \theta$ are candidate overlaps.]}

% Algorithm 5: Deterministic Schema Merge (sub-step)
\begin{algorithm}[t]
\caption{Deterministic Schema Merge via Stable Matching}
\label{alg:merge}
\begin{algorithmic}[1]
  \Require Validated shards $\{S_1,\ldots,S_k\}$, threshold $\theta$
  \Ensure Merged global schema $\mathcal{S}$, \texttt{MergeDecisionLog}
  \State \textit{[Placeholder: Pass 1---pairwise $\sigma = \alpha\,\sigma_{\text{name}} +
         (1{-}\alpha)\,\sigma_{\text{attr}}$ scoring (embedding cosine + lexical fallback;
         column-pair GS pass) and Gale--Shapley stable matching;
         Pass 2---FK consistency pre-validation per matched pair;
         Pass 3---column union + relationship remapping;
         post-merge repairs: junction consolidation, relationship name repair,
         FK type alignment, orphan FK wiring, connectivity check]}
\end{algorithmic}
\end{algorithm}

\subsection{Post-Merge Review and Domain Audit}
\label{subsec:schema-finalize}
\textit{[Para: After merging, a \textsc{MergeReviewer} LLM agent reviews the merged schema
$\mathcal{S}$, the \texttt{MergeDecisionLog}, and the original per-shard facts.
It identifies merge decisions that appear semantically inconsistent (e.g., two tables with
different semantics merged due to a high name-similarity score) and proposes targeted patches.]}

\textit{[Para: A \textsc{DomainAuditor} LLM loop agent then performs a deeper structural audit
using the \texttt{DomainIntelligenceReport}, the full fact clusters, and any structural
errors emitted by a deterministic patch validator.
It produces a \texttt{CritiqueReport} with typed patches (add table, rename column, add FK,
etc.); each patch is validated deterministically before being applied.
The loop continues until the \texttt{CritiqueReport} is empty or the retry budget is
exhausted.]}

\textit{[Para: Finally, a \textsc{ComplianceCertifier} LLM agent performs a
one-shot analytical-utility check---verifying that $\mathcal{S}$ can express the
analytical queries implied by $\mathcal{D}$ (e.g., that columns needed to compute
portfolio-level aggregates are present).
The certifier's verdict is informational; it does not block pipeline execution but
is logged for user inspection.
This produces the final validated global schema $\mathcal{S}$.]}

\caption{Stage 2: Schema Generation}
\label{alg:stage2}
\begin{algorithmic}[1]
  \Require Fact set $\mathcal{F}$
  \Ensure Global schema $\mathcal{S}$
  \State \textit{[Placeholder: DomainIntelligenceExtractor (one-shot LLM)

\section{Stage 3: Constraint Modeling}
\label{sec:stage3}
% ============================================================

\subsection{Structured Constraint Representation}
\textit{[Para: All constraints are stored in a nested structured format (JSON):
(a) univariate distributions (type, parameters, target column);
(b) decision-tree logical constraints (if--then rules over column values, including
cross-table aggregations such as SUM, MAX);
(c) multivariate distribution groups (\S\ref{subsec:multivariate}).
This representation enables downstream deterministic validation and code generation.]}

\subsection{Constraint Extraction and Feasibility Verification}
\textit{[Para: The global schema graph is sharded into connected components.
A constraint-modeling agent extracts structured constraints per component in parallel.
A mathematics agent then verifies distribution feasibility (e.g., checking that parameter
values are in the valid domain for the specified distribution family).
Decision-tree feasibility is checked deterministically (Algorithm~\ref{alg:dt-feasibility}).]}

% Algorithm 7: Deterministic Decision-Tree Feasibility Check (sub-step)
\begin{algorithm}[t]
\caption{Deterministic Decision-Tree Feasibility Check}
\label{alg:dt-feasibility}
\begin{algorithmic}[1]
  \Require Set of decision-tree constraints $T \subseteq \mathcal{C}$
  \Ensure Feasibility verdict per constraint; infeasibility report if any
  \State \textit{[Placeholder: per constraint---enumerate leaf conditions,
         check contradictory conjunctions (e.g., $x > 750$ AND $x < 300$),
         verify outcome type compatibility, check referenced columns exist in $\mathcal{S}$;

\subsection{Deterministic Inverse-Function Code for Single-Dependency Columns}
\label{subsec:det-codegen}
\textit{[Para: Key new contribution. For each column node $c \in \mathcal{G}$ with
\emph{exactly one} incoming dependency edge, Python code is synthesised
\emph{deterministically}---no LLM call is made.
The strategy is: (i)~synthesise an intermediate join or aggregate table satisfying the
aggregate constraint (the ``forward'' computation); (ii)~decompose back to base column
values using an inverse function whose form is fully determined by the aggregation type.
This guarantees correctness-by-construction for the covered columns and eliminates LLM API
cost for these nodes.
Covered aggregation types and their inverses: SUM $\to$ uniform partition,
MAX $\to$ argmax selection, AVG $\to$ constrained sampling around mean,
JOIN (FK assignment) $\to$ deterministic FK-key draw.]}

\subsection{LLM Ancestral Sampling for Intertwined Constraints}
\label{subsec:llm-sampling}
\textit{[Para: When a column node has multiple incoming dependency edges---i.e., its value
is governed by constraints involving several other columns simultaneously---no clean
closed-form inverse exists.
In this case, a code-generator LLM agent is invoked.
The agent receives the local sub-DAG centred on $c$, the relevant constraints from
$\mathcal{C}$, and the pre-determined generation order from $\mathcal{G}$.
It synthesises an \emph{inverse function} generatively: given already-sampled values for
the driver columns, it emits Python code that samples $c$ such that the constraints
are satisfied.
By following the reverse-causal order of $\mathcal{G}$, this scheme guarantees
constraints are satisfied \emph{by construction}, with no post-hoc repair.]}

\section{Discussion}
% ============================================================

\subsection{Limitations}
\textit{[Placeholder: (1)~Assumes NL input is internally consistent; no contradiction
detection or resolution.
(2)~Multivariate distribution support is partial---see \S\ref{subsec:multivariate};
copula or conditional-chain design not yet finalised.
(3)~Deterministic inverse synthesis covers only single-dependency columns;
many-to-many joins and compound multi-column aggregations fall back to LLM.
(4)~Context enrichment relies on web retrieval quality; domain jargon absent from
public knowledge bases may still be under-specified.]}

\subsection{Failure Modes}
\textit{[Placeholder: (a)~Overly ambiguous NL where enrichment cannot recover missing
distribution parameters (e.g., no public definition of a proprietary scoring model).
(b)~DAG cycles introduced by contradictory logical constraints that the patch agent
cannot resolve within the retry budget.
(c)~Distribution parameter hallucination for rare distribution families
(e.g., generalised extreme value distributions).
(d)~Smoke-test failures that cannot be repaired by the code-generation agent
within 5 retries---currently causes a pipeline abort.]}

\subsection{Future Directions}
\textit{[Placeholder: (1)~Multivariate distributions: copula-based joint sampling to model
cross-column correlations beyond univariate priors; Gaussian copulas as a first step.
(2)~Contradiction detection and resolution in NL input before fact extraction.
(3)~Extending deterministic code generation to multi-dependency chains
(iterative inverse composition for chained aggregations).
(4)~Schema evolution: incrementally updating the generated schema and program as the
NL description changes, without full pipeline re-execution.
(5)~Support for non-relational output (document stores, graph databases).]}
